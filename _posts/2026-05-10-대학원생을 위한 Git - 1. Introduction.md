## 들어가며

먼저 고백을 하나 하자면, **나는 개발자 출신이 아니다.**
학부에서는 기계공학을 전공했고, 컴퓨터공학을 따로 배운 적은 없다.
대학원에 들어와서야 C++을 본격적으로 만졌고, 지금도 git의 모든 기능을 능숙하게 쓴다고는 못한다.
다만 "비개발자 출신 대학원생이 git에서 어디에서 헤매는지" 정도는 누구보다 잘 안다고 자신한다.

이 시리즈를 시작하는 이유는 단순하다.
비슷한 처지의 후배 대학원생들에게 매번 같은 조언을 구두로 반복하고 있는 내 모습을, 어느 순간부터 자주 발견하게 됐기 때문이다.
이왕 할 얘기라면 글로 한 번 정리해두는 게 서로에게 편할 것 같았다.

다행히 그동안 [Patchwork++](https://github.com/url-kaist/patchwork-plusplus), [KISS-Matcher](https://github.com/MIT-SPARK/KISS-Matcher), [ERASOR](https://github.com/LimHyungTae/ERASOR) 같은 open-source 프로젝트를 운영해보고, [TEASER++](https://github.com/MIT-SPARK/TEASER-plusplus)와 [Hydra](https://github.com/MIT-SPARK/Hydra)에서는 co-maintainer로 일해보면서 git에 대한 감각이 어느 정도 잡혔다.
완벽하지는 않다.

그 간극을 메우는 데 이 시리즈가 작은 도움이 됐으면 한다.

## 첫 번째 메시지: 백업, 백업, 백업

소설가 김영하는 소설 쓰기에 제일 중요한거 하나만 알려달라는 학생의 질문에 '무조건 백업하라'라고 알쓸신잡에서 말한 일화가 있다.
컴퓨터가 한 번 죽었을 때 원고를 잃어본 사람만 그 무게를 안다, 같은 톤으로 말이다.
그래서 그는 클라우드에도 올리고, 외장하드에도 복제하고, 자기 자신에게 메일로도 보내둔다고 했다.
한 번 사라진 글을 "다시 쓴다"는 건 사실상 불가능하다는 걸 본인이 직접 겪어봤기 때문이다.

연구실 코드도 정확히 같다.
**한 번 날아간 코드는 "다시 짜는" 게 아니라 "다시 잃는" 시간이다.**

이 정서를 한 컷으로 압축한 게 개발자들 사이에 유명한 다음 밈이다.

![화재시 행동 요령 밈: git commit, git push, 그리고 대피](/img/in-case-of-fire.webp)

> **불 났을 때 행동 요령:**
> 1. `git commit`
> 2. `git push`
> 3. 건물에서 나간다

진지한 화재 대처법이 아니다.
이 밈이 진짜로 말하는 건 화재가 아니라 **습관**이다.
평소에 `commit`과 `push`가 얼마나 자주, 얼마나 반사적으로 일어나야 하면 "불나면 그것부터 한다"는 농담이 가능할까.
**당장 내일 컴퓨터가 망가지더라도 어제까지 작업한 내용은 어딘가에 살아남아 있어야 한다.** 이 시리즈의 첫 글이 전하고 싶은 메시지가 딱 그것이다.

### 대학원생들이 흔히 하는 실수

생각보다 많은 대학원생이 코드를 **자기 노트북 또는 워크스테이션에만** 둔다.
- ssh로 학교 서버에 접속해 짜놓은 코드
- 외장하드에만 백업해 둔 데이터셋 처리 스크립트
- "어차피 나만 쓸 거니까"라며 GitHub에 올리지 않은 실험 코드
- "지도교수님이 공개를 원하지 않으셔서 git 자체를 안 쓴다" (이건 git에 대한 큰 오해이다. 잠시 후에 풀겠다)

이 모든 게 **단 한 번의 사고**로 사라진다.
- 노트북 도난이나 분실
- SSD 사망 (예고 없이 죽는다)
- nvidia driver가 꼬여서 부팅 자체가 안 됨
- 카페에서 음료 쏟음
- 진짜로 화재나 침수
- 단순 실수로 `rm -rf`를 잘못 침

### 나도 똑같이 잃어봤다

세어보니 대학원 6년 동안 **Ubuntu 셋업/포맷을 어림잡아 10번 정도** 했다.
- 박사 입학 직후 워크스테이션 처음 셋업
- nvidia driver가 꼬여서 두세 번 다시 깖
- NAVER LABS 인턴 가서 새 컴퓨터 셋업
- StachnissLab(University of Bonn) 방문 학자 가서 새 컴퓨터 셋업
- 졸업하고 MIT에 postdoc으로 와서 또 셋업

그때마다 "이건 자잘한 거니까 굳이 안 올려도 되겠지" 했던 코드들이 사라졌다.
지금 와서 다시 짜면 더 잘 짜기야 하겠지만, 아무리 그래도 **사라진 코드는 사라진 시간이다.**
같은 시행착오를 두 번 겪는 비용은 결코 작지 않다.

### 그래서 git을 쓴다

git은 결국 **타임머신이자 보험**이다.
- `git commit`: 변경 내용을 한 번의 snapshot으로 저장한다.
- `git push`: 그 snapshot을 원격 서버(GitHub)에 백업한다.

이 두 줄을 습관으로 만들기만 해도 위에 나열한 사고들 중 절반 이상은 "별일 아닌 일"로 끝난다.

## "공개하기 싫은데요"라는 흔한 오해

git을 권할 때마다 자주 듣는 오해가 하나 있다.

> "GitHub에 올리면 다 공개되잖아요. 저는 그게 싫어요. 그래서 git을 안 써요."

**그렇지 않다.**
GitHub은 **private repository**(나만, 혹은 내가 초대한 사람만 볼 수 있는 저장소)를 무료로, 무제한 만들 수 있다.
대학원생이라면 [GitHub Student Developer Pack](https://education.github.com/pack)으로 추가 혜택까지 받을 수도 있다.
즉, **나만 보는 비공개 코드 백업 용도**로도 GitHub은 충분히 기능한다.

git을 쓰는 것과 코드를 공개하는 것은 **별개의 결정**이다.
- 일단 git은 쓴다 (백업 + history 관리)
- 공개 여부는 다음 단계의 문제이다 (다음 글에서 따로 다룬다)

이 둘을 분리해서 생각하는 것만으로도 git에 대한 심리적 장벽이 크게 낮아진다.

## 이 시리즈가 다루려는 것

비개발자 출신 대학원생, 특히 SLAM/robotics 분야 연구실에서 갓 코딩을 시작한 사람을 독자로 가정하고 쓴다.
다룰 주제는 다음과 같다.

1. **Introduction (이 글)**: 왜 git을 써야 하는가. 코드는 노트북이 아니라 cloud에 산다.
2. **Git은 메신저다**: 코드 공개라는 행위가 왜 연구의 일부가 됐는가, 그리고 지도교수 동의가 왜 우선인지.
3. **Patch적으로 사고하기**: commit과 PR을 의미 단위로 자르는 법, 협업 시 충돌 회피.
4. (이후 commit message 작성법, branch 전략, PR 문화, README 작성, secret/`.gitignore` 등 추가 예정)

각 글은 따로 읽어도 되게 썼지만, 가능하면 순서대로 읽기를 권한다.

## 정리

- 코드는 노트북이 아니라 **GitHub에 산다.** 노트북은 그저 임시 작업 공간이다.
- 사고는 예고 없이 온다. SSD 사망, 도난, 포맷, `rm -rf` 한 번이면 끝이다.
- 김영하 작가가 원고를 위해 백업에 강박을 가졌듯, 연구자도 코드를 위해 `commit`과 `push`에 강박을 가져야 한다.
- git ≠ 코드 공개. private repo로도 충분하다.
- 일단 `git commit`, `git push`. 이 두 줄만으로도 보험에 든 셈이다.

다음 글에서는 git이 단순한 보험을 넘어 **메신저**로 작동하는 방식, 그리고 그게 본인의 연구에 어떤 의미를 갖는지를 다룬다.
